# 末日值班表 v2 - 重设计方案

## 1. 重设计目标

这次重设计不解决“内容不够多”的问题，而是解决 3 个更根本的问题：

1. 过程枯燥  
   现在的主循环接近“读事件 -> 点选项 -> 看数值变化”，缺少递进压力与延迟后果。

2. 结局单一  
   目前结局主要由数值归零触发，路线结局也偏概念化，玩家很难感到“这是我自己打出来的人生”。

3. 缺少真实感与代入感  
   现有系统更像抽象资源管理，缺少“队伍关系、承诺、伤病、负债、舆论、后悔”这些真正构成末日求生体验的东西。

新的设计目标是：

- 每次选择都至少影响 2 个维度：资源、关系、名声、风险、路线承诺。
- 玩家不是在“刷事件池”，而是在一个会记仇、会追债、会连锁反应的环境里生存。
- 结局不只区分死没死，还区分你变成了什么样的人，以及你把别人带到了哪里。

## 2. 新的一句话卖点

你不是救世主，而是灾变后仍被迫排班、分配物资、安抚同伴、做脏决定的普通人。活下来不难，难的是活下来以后，你还剩下什么。

## 3. 核心体验原则

### 3.1 选择必须有账单

所有关键选择都至少包含以下两层之一：

- 即时收益：拿到药、守住据点、换到情报、平息争吵
- 延迟账单：有人记恨你、感染恶化、据点失去功能、某 NPC 改变立场、某路线被锁死

### 3.2 风险必须具体

不再使用“泛风险”表达一切，风险拆成可识别、可预判的类型：

- 尸潮风险：由噪音、路线暴露、夜间活动决定
- 人类冲突风险：由名声、承诺违约、资源分配不公决定
- 内部崩溃风险：由疲劳、伤病、低信任、长期压抑决定

### 3.3 角色必须有记忆

NPC 不能只是关系值条目，而要记录玩家做过的关键行为：

- 是否救过他
- 是否利用过他
- 是否把风险转嫁给平民
- 是否兑现承诺
- 是否在公开场合羞辱/支持过他

关系变化不再只是 `+8 / -10`，而是进入后续事件判断。

### 3.4 结局必须回答 3 个问题

- 你活下来了吗？
- 谁还愿意和你站在一起？
- 你是靠什么方式活下来的？

## 4. 新核心循环

每回合循环改为：

1. 结算上一轮的延迟后果
2. 播报“今晚/今天最危险的 3 个问题”
3. 触发当前事件
4. 玩家做选择
5. 立刻应用即时结果
6. 将后续代价放入延迟队列
7. 更新 NPC 记忆、承诺状态、据点模块状态
8. 推进到下一天并执行日常消耗
9. 检查是否触发危机链、角色链、结局链

体验重点从“本回合赚了多少”改为“我是不是在把未来卖掉”。

## 5. 新状态系统

状态不求多，但必须互相作用。建议保留 8 个主状态，并新增 4 个结构状态。

### 5.1 主状态（玩家直观看到）

1. `health` 健康  
   肉体状况。受伤、感染、极端饥饿会长期压低。

2. `infection` 感染  
   不是立刻死亡条，而是影响被接纳程度、工作效率、剧情分支。

3. `hunger` 饥饿  
   高饥饿压低体能和判断力，恶化内部矛盾。

4. `fatigue` 疲劳  
   替代当前 `stamina` 的纯体能表达。疲劳高时夜间失误率上升。

5. `supplies` 物资  
   包括食物、药、工具、电池、净水片的综合能力。

6. `morale` 士气  
   队伍是否还能继续相信“明天值得活”。低士气会放大冲突与逃离事件。

7. `trust` 互信  
   你和队伍之间的互信，不等于单个 NPC 好感。

8. `threat` 外部威胁  
   对据点的总体压迫，来源于噪音、暴露度、失败行动、邻近失守。

### 5.2 结构状态（系统内部主导连锁）

1. `wounds` 伤口数 / 开放伤口标记  
   受伤后若未处理，会持续带来感染和失误。

2. `debt` 承诺债务  
   你答应的事没做、救援没去、分配没兑现，都会形成“债”。

3. `reputation` 外部名声  
   其他据点如何看你。决定交易、合作、收容、冲突事件。

4. `shelterModules` 据点模块  
   将“据点稳固”拆为模块：
   - 医疗角
   - 电力
   - 水源
   - 防线
   - 广播/情报

模块比总值更真实，也更容易写出具体事件。

## 6. 新的世界结构

场景不再像随机城市切片，而是围绕 4 个长期生存板块运转：

1. 据点治理  
   配给、守夜、隔离、照护、纪律、轮班、争执

2. 外出行动  
   搜刮、救援、侦察、交易、医疗、修路、转运

3. 人际政治  
   NPC 联盟、质疑、挑战领导权、秘密交易、举报、告密

4. 大环境推进  
   尸潮迁移、断电、雨天、疫病扩散、外部广播、撤离窗口、周边据点沦陷

玩家每一天都应该能感觉到：

- 自己不是在单独冒险
- 据点不是背景板
- 世界在自己选择之外也会继续变化

## 7. 事件系统重构

事件池改为 4 类并行系统。

### 7.1 危机主线链

少而重，负责拉出整局节奏。

建议主线阶段：

1. 断联与收容
2. 医疗与隔离争议
3. 配给制度与秩序建立
4. 外部势力接触
5. 撤离窗口或长期固守抉择

主线链每隔数天推进一次，不靠纯随机抽到。

### 7.2 NPC 关系链

每个核心 NPC 都有：

- 需求
- 底线
- 对玩家的关键记忆
- 个人崩溃节点
- 最终立场归宿

建议保留 5 到 6 个核心 NPC，但每个必须能“背叛 / 离开 / 深度绑定 / 替你承担 / 拒绝服从”。

### 7.3 日常生存链

负责制造真实感与微观压力。

例如：

- 夜班时有人偷吃配给
- 医疗角药品过期
- 有小孩发烧但抗生素不够
- 外出队员不愿再跑
- 一位平民请求加入但会拖慢行动

这类事件不一定决定结局，但决定玩家对这局人生的感受。

### 7.4 追债事件链

这是 v2 的关键。

当玩家做出某些选择时，不直接结束，而是生成未来事件：

- 放弃救援 -> 两天后家属质问
- 公开偏袒 -> 一周后内部派系形成
- 透支外出队 -> 后续任务成功率下降
- 对外隐瞒感染 -> 后续引发信任雪崩

这条链让游戏从“选项题”变成“承担后果”。

## 8. 选择设计规范

每个事件尽量保持 3 个选项，不追求 5 到 6 个表面丰富的弱选项。

推荐的 3 类选项结构：

1. 稳妥但代价高  
   消耗资源，保住秩序或关系

2. 高收益高风险  
   可能赚很多，但会提高威胁或形成延迟债务

3. 冷血或现实主义  
   当下效率最高，但伤害名声、人性、互信或特定 NPC

每个选项必须写清楚：

- 即时效果
- 潜在后果类型
- 是否不可逆

不要求直接公开数值，但要公开风险语义。

## 9. 新结局体系

结局从单点触发改为组合判定。

### 9.1 生存结局层

- 活着撤离
- 活着固守
- 活着流亡
- 死于耗尽
- 死于感染
- 死于内部清算

### 9.2 关系结局层

- 仍有人信任你
- 队伍虽然活着但彻底分裂
- 你被放逐
- 你成为临时领袖
- 你只剩一个人

### 9.3 人格结局层

- 守住底线
- 现实主义妥协
- 冷血统治
- 自我牺牲
- 麻木求生

最终结局文案由三层拼装，形成真正有差异的收束。

示例：

- 生存层：活着固守
- 关系层：成为临时领袖
- 人格层：冷血统治

输出结局：

“你守住了据点，但据点里的人不再把你当同伴，而是把你当规则本身。”

## 10. 新开局设计

开局不要再从“随机身份微加成”开始，而要让玩家立刻进入处境。

推荐开场：

- 第 0 夜，楼内断联
- 一名伤者、一名情绪失控者、一条真假不明的撤离广播同时出现
- 玩家必须先做第一个价值排序：
  - 先救伤者
  - 先封门稳住局面
  - 先去抢通信设备

这个开局能立刻定义玩家风格。

职业系统仍可保留，但用途改为“打开处理方式”，不是简单加减数值。

例如：

- 运维支持：能抢修电力或广播
- 内容策划：能在公开场合安抚群众
- 客户运营：能调停冲突、降低承诺债务
- 设计师：能快速制作警示与标识，强化秩序效率

## 11. UI 重构方向

当前 UI 视觉层不差，但信息结构仍偏“展示型”。v2 要改成“指挥型界面”。

### 11.1 主屏三段式

1. 当前现场  
   标题、正文、关键人物、地点、时间段

2. 今晚最危险的 3 件事  
   例如：
   - 医疗角断药
   - 外墙噪音过高
   - 乔楠正在煽动质疑配给

3. 选择区  
   每个选项附带风险标签：
   - 高噪音
   - 高关系代价
   - 延迟后果
   - 不可逆

### 11.2 右侧信息栏

- 队伍状态：互信、士气、疲劳、伤员数
- 据点模块：水、电、医、防、广播
- 关键人物：当前最支持你和最不信任你的人
- 追债列表：还有哪些承诺没处理

### 11.3 日志改造

日志不再只记“发生了什么”，而要记：

- 谁记住了什么
- 哪个债务被创建
- 哪个模块恶化或修复
- 哪条路线已封死

## 12. 内容生产策略

不要一开始就追求 80 个事件。先做可打通、可复玩、会追债的前 7 天内容。

### 12.1 首批内容范围

- 6 个主线事件
- 12 个日常事件
- 10 个 NPC 事件
- 8 个追债事件
- 6 个结局模板

### 12.2 每个事件至少有一个“回响”

所谓回响，是指它会在之后至少一次被提及或兑现。

例如：

- 你第 2 天没救的人，第 5 天的家属会来找你
- 你第 3 天建立的轮班表，第 6 天会决定谁支持你

只要做到这一点，内容量不大，体验也会立起来。

## 13. 对现有代码的落地改造

本项目是纯静态网页，可在不引入框架的前提下完成升级。建议按以下顺序改。

### 13.1 `data/events.js`

重构重点：

- 事件数据拆成：
  - `storyEvents`
  - `npcEvents`
  - `dailyEvents`
  - `debtEvents`
- NPC 数据增加：
  - `memoryKeys`
  - `loyalty`
  - `fear`
  - `personalGoal`
- 据点模块单独定义：
  - `water`
  - `power`
  - `medical`
  - `defense`
  - `intel`
- 结局数据改为多层模板，不再只有固定失败结局

### 13.2 `engine.js`

必须新增的系统：

1. `delayedConsequences`  
   存储延迟结算事件：
   - `dueDay`
   - `type`
   - `sourceChoice`
   - `payload`

2. `npcMemory`
   存储每个 NPC 对关键行为的记忆标签。

3. `debts`
   存储玩家对外承诺与未完成事项。

4. `shelterModules`
   替代单一 `shelter` 的部分表达。

5. `dailyBriefing`
   每回合先根据当前状态生成“今日/今晚风险播报”。

6. `endingAssembler`
   根据生存、关系、人格 3 层条件拼装最终结局。

现有可复用部分：

- 存档读写结构
- 事件抽取和模板化工具函数
- 基础效果应用逻辑

需要弱化或删除的部分：

- 单纯按类别加权的事件权重思路
- 过于泛化的行动检定结果
- 只靠数值阈值直接触发的大部分结局

### 13.3 `ui.js`

重构重点：

- 顶部增加今日风险播报
- 右侧改为队伍 / 据点 / 追债三区块
- 关系网不只显示数值，还要显示立场标签
- 选项按钮增加风险标签与后果提示
- 日志分为“事件记录”和“记忆记录”

### 13.4 `index.html`

结构上建议新增：

- `briefing-panel`
- `shelter-panel`
- `debt-panel`
- `consequence-panel`

现有主骨架可以保留，不需要推翻页面结构。

## 14. 开发顺序建议

为了尽快把体验拉起来，按下面顺序开发：

1. 先改引擎数据结构  
   目标：能支持延迟后果、NPC 记忆、债务

2. 再做前 7 天内容  
   目标：一局 20 分钟内就能感受到“这次选择会追上我”

3. 最后改 UI  
   目标：让玩家看懂自己究竟在失去什么，而不是只看见数字

## 15. 第一版可交付标准

满足以下条件，就说明 v2 已经脱离旧版本的根本问题：

1. 玩家在前 5 天内至少遇到 2 次“过去选择的回响”
2. 至少有 3 名 NPC 会因玩家行为改变长期立场
3. 至少有 2 个据点模块会独立恶化/修复
4. 玩家能在 UI 上明确看到未处理承诺和近期风险
5. 至少能打出 6 个有明显人格差异的结局文本

如果以上 5 条做不到，说明这次重设计仍停留在“文案换皮”，而不是玩法升级。
